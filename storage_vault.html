Storage — KMS Vault, Object Bucket & ATP Database

Creates the encryption key vault, stores credentials as secrets, provisions the object storage landing zone, and deploys the private Autonomous Transaction Database.

ⓘ Prerequisites

Complete Step 1 & 2 (Networking) first. The ATP Database must be placed in the sod-private-subnet created in the previous page. Complete Step 3 entirely before Step 4 — the ATP database is configured to use the KMS encryption key created in Step 3.2, so the key must exist first.

Step 3 — Create KMS Vault, Master Key & Secrets

The KMS (Key Management Service) Vault is a Hardware Security Module (HSM) — a tamper-proof physical device inside Oracle's datacentre that stores your encryption keys. Once a key is created inside an HSM vault, it never leaves the hardware. Any application that needs to decrypt data must call the vault API — the key itself is never exposed. You'll use this vault to store two critical secrets: the ATP database admin password and the HCM integration API credentials. Functions retrieve these secrets at runtime using Resource Principals (no static passwords in code).

STEP 3.1
Identity & Security Vault Create Vault

What you're doing: Provisioning the HSM vault container. The DEFAULT vault type uses a shared HSM partition — fully HSM-backed but lower cost than a dedicated partition. This vault is the root trust anchor for all sensitive credentials in the platform. Everything — the database password, API keys — flows through here.

How to do it: Navigate to Identity & Security → Vault and click Create Vault.

Vault Name
sod-kms-vault
Vault Type
DEFAULT
STEP 3.2
sod-kms-vault Master Encryption Keys Create Key

What you're doing: Creating the Master Encryption Key (MEK) that wraps every secret stored in this vault. AES-256 (32 bytes) is the industry-standard symmetric encryption algorithm — the same used by banks and governments. HSM protection mode means the key material lives exclusively inside the hardware module; Software protection mode would store it in software (less secure). Always use HSM for production environments.

How to do it: Open sod-kms-vault → Master Encryption Keys and click Create Key.

sod-encryption-key
Protection Mode
HSM
Algorithm

Key Length
256 Bits (32 Bytes)
STEP 3.3
sod-kms-vault Secrets Create Secret — DB Password

What you're doing: A Secret is a versioned, encrypted blob stored inside the vault and wrapped with the master key from Step 3.2. When a Function needs the DB password, it calls the Vault API to retrieve this secret — the actual password value is never stored anywhere in the function code, environment variables, or configuration files. OCI automatically rotates secrets when you create new versions. Paste the actual DB admin password you intend to use as the Secret Contents value.

How to do it: Click Secrets in the vault sidebar, then Create Secret.

STEP 3.4
sod-kms-vault Secrets Create Secret — HCM Credentials

What you're doing: Storing the HCM system API credentials (e.g. Workday Client ID + Client Secret, or Oracle HCM OAuth token) as a second secret. Store these as a JSON object so a single vault call retrieves all credentials needed by the connector function. Example payload: {"client_id": "xxx", "client_secret": "yyy", "token_url": "https://..."}. This means rotating HCM credentials is a vault operation — no code redeployment needed.

How to do it: Repeat the same process as Step 3.3 with the values below.

Why We Do It

Storing credentials inside an HSM vault means no passwords exist in source code or function environment variables. Functions retrieve them at runtime using Resource Principals — no static API keys required.

Step 4 — Create Object Storage Bucket & ATP Database

The Object Storage Bucket is the entry point for all inbound audit data — client applications upload files here. By enabling Object Events, the bucket automatically fires an OCI Event every time a file is uploaded, which triggers the processing pipeline. The ATP Database (Autonomous Transaction Processing) is Oracle's fully-managed, self-tuning database — it automatically handles patching, backups, and performance tuning. Running it in private endpoint mode means only resources inside the private subnet can connect to it directly.

STEP 4.1
Storage Object Storage & Archive Storage Create Bucket

What you're doing: Creating the landing zone where all client audit files (CSV/JSON role and access exports) are deposited. The critical setting here is Emit Object Events — without this checkbox, file uploads happen silently and nothing downstream gets triggered. With it enabled, every file upload fires an Object - Create event to the OCI Events Service, which in Step 9 you'll wire to automatically invoke the sod-enqueuer function. This is the starting point of the entire ingestion pipeline.

How to do it: Navigate to Storage → Buckets and click Create Bucket. Make sure to check Emit Object Events before saving.

Bucket Name
landing-zone-assessment

Emit Object Events

STEP 4.2
Oracle Database Autonomous Database Create Autonomous Database

What you're doing: Provisioning the central ATP database that stores all SoD assessment data — employee records, role assignments, audit runs, findings, and exceptions. Choosing Transaction Processing (OLTP) workload type optimizes the DB for high-concurrency small row inserts/updates (which is what the function loaders do). Setting Private Endpoint Only means the database gets no public IP and cannot be reached from outside the VCN. Only the serverless functions and the DB bootstrap scripts running inside the private subnet can connect to it.

How to do it: Navigate to Oracle Database → Autonomous Database and click Create Autonomous Database. Under Network Access, select Private Endpoint Only.
