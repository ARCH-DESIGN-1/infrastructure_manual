Storage — KMS Vault, Object Bucket & ATP Database

Creates the encryption key vault, stores credentials as secrets, provisions the object storage landing zone, and deploys the private Autonomous Transaction Database.

ⓘ Prerequisites

Complete Step 1 & 2 (Networking) first. The ATP Database must be placed in the sod-private-subnet created in the previous page. Complete Step 3 fully before Step 4 — the ATP database uses the KMS encryption key.

Step 3 — Create KMS Vault, Master Key & Secrets

Provision an HSM-partitioned Key Management Vault, generate a master AES encryption key, then store database passwords and HCM API credentials as encrypted secrets.

STEP 3.1
Identity & Security Vault Create Vault

Navigate to Identity & Security → Vault and click Create Vault. Select DEFAULT type for HSM-partitioned hardware security.

Vault Name
sod-kms-vault
Vault Type
DEFAULT
STEP 3.2
sod-kms-vault Master Encryption Keys Create Key

Open sod-kms-vault, go to Master Encryption Keys and click Create Key. This key wraps all secrets stored in this vault.

Key Name
sod-encryption-key
Protection Mode
HSM
Algorithm
AES
Key Length
256 Bits (32 Bytes)
STEP 3.3
sod-kms-vault Secrets Create Secret — DB Password

Go to Secrets tab in the vault and click Create Secret. Store the ATP Database admin password. Paste the actual password value in the Secret Contents field.

Secret Name
sod-db-password
Encryption Key
sod-encryption-key
Secret Type Template
Plain-Text
STEP 3.4
sod-kms-vault Secrets Create Secret — HCM Credentials

Create a second secret to store HCM integration API credentials (e.g. Workday client ID and client secret as a JSON payload).

Secret Name
sod-hcm-credentials
Encryption Key
sod-encryption-key
Secret Type Template
Plain-Text
Why We Do It

Storing credentials inside an HSM vault means no passwords exist in source code or function environment variables. Functions retrieve them at runtime using Resource Principals — no static API keys required.

Step 4 — Create Object Storage Bucket & ATP Database

Provision the landing zone bucket (where audit files are uploaded) and the private Autonomous Transaction Processing database (where assessment data is stored and evaluated).

STEP 4.1
Storage Object Storage & Archive Storage Create Bucket

Navigate to Storage → Buckets and click Create Bucket. Enable Emit Object Events — this is what triggers the ingestion function automatically on upload.

Bucket Name
landing-zone-assessment
Storage Tier
Standard
Emit Object Events
✓ Enable (check the checkbox)
STEP 4.2
Oracle Database Autonomous Database Create Autonomous Database

Navigate to Oracle Database → Autonomous Database and click Create Autonomous Database. Set Network Access: Private Endpoint Only and bind it to the private subnet.

sod-assessment-adb
Database Name
sodassessdb
Workload Type

Network Access

VCN

Subnet

Admin Password

✓ Checkpoint — Storage & Vault Complete

At this point you should have: 1 KMS Vault with 1 Key and 2 Secrets, 1 Object Storage Bucket (with events enabled), and 1 ATP Database (private endpoint). Proceed to Step 5 on the next page.
