API Gateway — Public Endpoint & Route Deployment

Deploys a public API Gateway into the public subnet, then maps four REST endpoint routes to their corresponding private serverless backend functions.

ⓘ Prerequisites

All 7 functions from Step 7 must already be deployed in sod-app before configuring routes. The gateway routes reference function names directly.

API Route Mapping Reference

All routes share the base path prefix /sod/v1. Final URL format: https://<gateway-hostname>/sod/v1/<endpoint>

Route Path Method Backend Function Purpose
/ingest POST sod-api-receiver Receives audit file uploads from enterprise clients
/connectors/run POST sod-manual-trigger Triggers an on-demand HCM connector sync job
/runs/{runId}/status GET sod-run-status Returns current pipeline status for a given run
/connectors GET sod-list-connectors Lists available HCM integration connectors

Step 8 — Create Gateway, Deployment & Routes

The API Gateway is the single public-facing entry point for the entire SoD platform. It lives in the public subnet, accepts HTTPS requests from clients anywhere on the internet, authenticates and routes them to the correct serverless function in the private subnet, and returns the response. A Deployment is the versioned configuration that holds all the route definitions. Routes map a specific URL path + HTTP method combination to a backend function. This design means clients never talk directly to functions — all traffic goes through the gateway, which can enforce rate limits, CORS, and authentication policies.

STEP 8.1
Developer Services API Management Create API Gateway

What you're doing: Creating the public-facing API Gateway instance. Selecting Public type means OCI assigns the gateway a public IP address reachable from the internet. Placing it in sod-public-subnet connects it to the public route table (which has the Internet Gateway route), so inbound HTTPS traffic from clients can reach it. The gateway acts as a reverse proxy — it accepts the request, checks the routes you define in Step 8.2, and forwards it to the matching serverless function over the private network.

How to do it: Navigate to Developer Services → API Management → Gateways and click Create Gateway.

STEP 8.2
sod-api-gateway Deployments Create Deployment

What you're doing: A Deployment is the configuration object that holds all your route definitions. It's versioned — if you need to add or change routes in the future, you create a new deployment. The Path Prefix /sod/v1 is prepended to all route paths, so clients call https://<gateway-host>/sod/v1/ingest, not just /ingest. The v1 prefix means you can add a v2 deployment in the future with breaking changes without disrupting existing clients.

How to do it: Open sod-api-gateway → Deployments and click Create Deployment. You can add routes within the same wizard — don't click Create until all 4 routes are added.
