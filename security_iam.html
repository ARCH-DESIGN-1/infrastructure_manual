IAM — Dynamic Groups & Security Policies

Creates dynamic resource groups that identify serverless functions and database instances at runtime, then grants them least-privilege access to secrets, object storage, and queues via IAM policies.

ⓘ Prerequisites

Have your Compartment OCID ready — copy it from Identity & Security → Compartments → [your compartment] → OCID. You will need to paste it into the dynamic group matching rules below.

Step 6 — Create Dynamic Groups & IAM Policies

OCI Identity & Access Management (IAM) uses a least-privilege model — by default, every resource can do nothing. You must explicitly grant permissions. Dynamic Groups are special IAM groups that automatically include OCI resources (Functions, Databases) that match a matching rule. Instead of managing per-resource credentials, you put resources in a Dynamic Group and grant the group permissions via a Policy. When a Function runs, it proves its identity using a Resource Principal — OCI knows which function is calling based on its OCID, and checks if its Dynamic Group has the necessary policy statements.

STEP 6.1
Identity & Security Dynamic Groups Create Dynamic Group — Functions

What you're doing: Creating a Dynamic Group that automatically includes every OCI Function deployed in your compartment. The matching rule uses resource.type = 'fnfunc' to match all Functions, scoped to your specific compartment by its OCID. This group is then granted permissions in Step 6.3 to: read vault secrets (to get the DB password), manage object storage files (to download audit uploads), and use queues (to enqueue/dequeue messages). Without this group, functions would get permission-denied errors when they try to do any of those things.

How to do it: Navigate to Identity & Security → Dynamic Groups and click Create Dynamic Group. Replace the placeholder with your actual Compartment OCID.

Name
sod-functions-dg
Matching Rule (replace <compartment_ocid>)
ALL {resource.type = 'fnfunc', resource.compartment.id = '<compartment_ocid>'}
STEP 6.2
Identity & Security Dynamic Groups Create Dynamic Group — Database

What you're doing: Creating a second Dynamic Group for the ATP Database instance. The database uses Oracle's DBMS_CLOUD and DBMS_SCHEDULER packages to read files from Object Storage and enqueue messages. For these internal database operations to succeed (when initiated from a stored procedure or package), the database itself needs to be a recognized resource principal. Without this group, database attempts to read from the bucket or write to a queue will be rejected.

How to do it: Create a second Dynamic Group with the rule below.

Name
sod-adb-dg
Matching Rule (replace <compartment_ocid>)
ALL {resource.type = 'autonomousdatabase', resource.compartment.id = '<compartment_ocid>'}
STEP 6.3
Identity & Security Policies Create Policy — Functions Policy

What you're doing: Creating an IAM Policy that grants the Functions Dynamic Group the specific permissions it needs. Each policy statement uses the format Allow [who] to [action] [resource type] in [scope]. Statement 1 allows functions to read vault secrets (for DB passwords). Statement 2 allows functions to upload/download files from Object Storage. Statement 3 allows functions to put/get messages on queues. Statement 4 allows the Events Service to invoke functions (needed for the bucket upload trigger). Statement 5 allows the API Gateway to invoke functions (needed for the public API routes).

How to do it: Navigate to Identity & Security → Policies, click Create Policy, enable Show manual editor, and paste all 5 statements.

Policy Name
sod-functions-policy
Policy Statements (5 statements)
Allow dynamic-group sod-functions-dg to read secret-bundles in compartment id <compartment_id> Allow dynamic-group sod-functions-dg to manage objects in compartment id <compartment_id> Allow dynamic-group sod-functions-dg to use queues in compartment id <compartment_id> Allow service cloudevents to use functions-family in compartment id <compartment_id> Allow any-user to use functions-family in compartment id <compartment_id> where request.principal.type = 'apigateway'
STEP 6.4
Identity & Security Policies Create Policy — Database Policy

What you're doing: Creating a second policy for the ATP Database Dynamic Group. Statement 1 allows the database to read objects from the landing zone bucket — required when a stored procedure calls DBMS_CLOUD.COPY_DATA to load CSV files. Statement 2 allows the database to put messages onto queues — required when a PL/SQL package needs to trigger an asynchronous connector job. Both are database-initiated operations that use Resource Principals to authenticate.

How to do it: Create another policy with 2 statements.

Policy Name
sod-adb-policy
Policy Statements (2 statements)
Allow dynamic-group sod-adb-dg to read objects in compartment id <compartment_id> Allow dynamic-group sod-adb-dg to use queues in compartment id <compartment_id>
✓ Checkpoint — IAM Complete

You should have 2 Dynamic Groups (sod-functions-dg, sod-adb-dg) and 2 IAM Policies (sod-functions-policy, sod-adb-policy). Proceed to Step 7 to deploy the containerized functions.
