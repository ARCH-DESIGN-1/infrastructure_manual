Phase 4

IAM Dynamic Groups & Policies Setup

Configures the compartment security policies to assign access dynamically using Resource Principals.

Step 6 — Establish Access Control Policies

Configure dynamic resource groupings to bind least-privilege security policy definitions directly to runtimes.

STEP 6.1
Identity & Security Dynamic Groups Create Dynamic Group

Group compartment serverless functions and database instances dynamically using rules.

Group 1: Serverless Functions Dynamic Group
Name: sod-functions-dg | Matching Rule: ALL {resource.type = 'fnfunc', resource.compartment.id = '<compartment_ocid>'}
Group 2: Autonomous Database Dynamic Group
Name: sod-adb-dg | Matching Rule: ALL {resource.type = 'autonomousdatabase', resource.compartment.id = '<compartment_ocid>'}
STEP 6.2
Identity & Security Policies Create Policy

Create access control policies allowing Dynamic Groups to invoke compartment resources securely.

Policy 1: sod-functions-policy Statements
Allow dynamic-group sod-functions-dg to read secret-bundles in compartment id <compartment_id> Allow dynamic-group sod-functions-dg to manage objects in compartment id <compartment_id> Allow dynamic-group sod-functions-dg to use queues in compartment id <compartment_id> Allow service cloudevents to use functions-family in compartment id <compartment_id> Allow any-user to use functions-family in compartment id <compartment_id> where request.principal.type = 'apigateway'
Policy 2: sod-adb-policy Statements
Allow dynamic-group sod-adb-dg to read objects in compartment id <compartment_id> Allow dynamic-group sod-adb-dg to use queues in compartment id <compartment_id>
Why We Do It

Assigning policies to dynamic groups allows functions and database runtimes to communicate securely using OCI Resource Principals, bypassing the need to generate, store, and rotate static API access keys.
