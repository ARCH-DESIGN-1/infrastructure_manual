SoD Platform Transaction Flows

Select a tab below to inspect the detailed connection layout for each transaction flow. We split these apart to prevent overlapping lines and showcase the decoupled, event-driven design.

Ingestion Sequence Description

1
Inbound Upload Request
The client application POSTs audit file metadata and payload to the public API Gateway entrypoint /sod/v1/ingest.
2
Function Gateway Execution
The API Gateway validates routing parameters and routes the request to sod-api-receiver running inside the private subnet.
3
Raw Land Write
The function streams the audit data payload to the secure Object Storage bucket landing-zone-assessment via the internal Service Gateway.
4
Object Event Trigger
The bucket fires a native OCI Object Create event. OCI Events intercepts this and evaluates the matching rules (bucketName filter).

Ingestion Sequence Description (cont.)

5
Enqueuer Function Call
OCI Events invokes sod-enqueuer which parses the event JSON to determine the uploaded file name, size, and bucket location.
6
Ingestion Queue Dispatch
The enqueuer writes a processing message containing this bucket key metadata onto sod-ingestion-queue, decoupling the pipeline.
7
Worker Queue Poll
The background processor daemon sod-processor polls the queue, hides the message, and downloads the file from Object Storage.
8
Database Bulk Staging
The processor function parses the CSV/JSON rows and streams them directly into staging tables inside the Autonomous Transaction Database (ATP).

HCM Connector Sync Flow

1
Trigger Ingress (Manual or Scheduled)
A connector sync is started either manually via the API Gateway endpoint POST /connectors/run or automatically by an internal database DBMS scheduler cron job.
2
Job Enqueued
The execution metadata is formatted as a JSON job payload and written to the sod-connector-queue. This prevents direct HTTP timeouts.
3
Worker Job Dequeue
The dedicated worker function sod-connector-processor polls the connector queue and picks up the job.

HCM Connector Sync Flow (cont.)

4
Secret Fetch & Auth
The connector function uses OCI Resource Principals to fetch the encrypted HCM credentials block (OAuth client ID/secret JSON) from the KMS vault.
5
Outbound API Egress
The function routes an HTTPS request out of the private subnet through the NAT Gateway to hit the external HCM platform (Workday, SuccessFactors, etc.), downloads the role access files, and saves them to Object Storage (which triggers the Ingestion flow).

Status / Metadata Flow

1
Client HTTP GET Request
The dashboard portal fires an asynchronous GET request to check processing status for a specific upload run (e.g. GET /runs/run-771/status) or retrieve available HCM connector options.
2
API Gateway Routing
The API Gateway intercepts the traffic, maps the route rules, and forwards the query downstream to the matching microservice function.
3
Database Staging Check
The function executes a lightweight SQL SELECT query against the private ATP database endpoint and returns the JSON result payload back to the gateway.

Transaction Rationale

Read-Write Separation: By splitting status checks and metadata retrieval into dedicated, lightweight microservices (sod-run-status, sod-list-connectors), the main heavy data ingestion engine (sod-processor) can load thousands of audit rows without being blocked by client dashboard read requests.

Resource Scale: These read-only status functions are assigned low memory thresholds (256MB) and low timeout settings (30s) to guarantee fast startup times and scale to high concurrency under operator dashboard updates.

Subnet Architecture Isolation

sod-public-subnet: Serves as the DMZ boundary. The public API Gateway acts as the secure entry gateway, isolating all internal workloads from direct public access.

sod-private-subnet: Hosts the compute resource instances (functions) and database nodes. Security rule constraints prohibit allocating public IP addresses inside this zone, ensuring zero ingress paths from external networks.

Gateway Routing Topologies

sod-internet-gateway: Routes inbound HTTPS requests into the public subnet routing group.

sod-nat-gateway: Provides private resources with an outbound-only path to talk to external cloud platforms.

sod-service-gateway: Forwards traffic to Oracle-managed backend infrastructure directly over internal backbone links, avoiding standard WAN links.
