Step-by-Step Connection Reference

Every connection in the diagram mapped to its configuration step:

Step From Connection / Route To Why
Step 1.1 OCI Tenancy Creates VCN 10.0.0.0/16 sod-network-vcn Master network boundary. All resources live inside this CIDR space.
Step 1.2 sod-network-vcn Attaches Internet Gateway sod-internet-gateway Enables inbound HTTPS traffic from internet clients to reach the API Gateway.
Step 1.3 sod-network-vcn Attaches NAT Gateway sod-nat-gateway Allows private subnet resources (Functions, DB) to make outbound calls to external HCM APIs without a public IP.
Step 1.4 sod-network-vcn Attaches Service Gateway sod-service-gateway Routes traffic to OCI-managed services (Object Storage, Queue, Vault) over Oracle's private backbone — no public internet.
Step 2.1 sod-public-route-table Route rule: 0.0.0.0/0 → Internet Gateway sod-internet-gateway Any outbound internet traffic from the public subnet exits through the Internet Gateway.
Step 2.2 sod-private-route-table Rule 1: 0.0.0.0/0 → NAT GW
Rule 2: All Oracle Services → Service GW		 sod-nat-gateway + sod-service-gateway Private resources get two exit paths: internet via NAT, Oracle services via Service GW.
Step 2.3 sod-network-vcn Creates public subnet, binds to sod-public-route-table sod-public-subnet (10.0.1.0/24) Only the API Gateway lives here. Has internet access, receives inbound client requests.
Step 2.4 sod-network-vcn Creates private subnet, binds to sod-private-route-table sod-private-subnet (10.0.2.0/24) Functions + Database live here. No public IPs, protected from inbound internet access.
Step 3.1 OCI Tenancy Creates HSM Vault sod-kms-vault HSM-backed secret storage. Functions retrieve credentials at runtime — no passwords in code.
Step 3.2 sod-kms-vault Creates AES-256 master key sod-encryption-key Wraps all secrets stored in the vault. Key lives in HSM hardware — never exposed.
Step 3.3 / 3.4 sod-kms-vault Stores encrypted secrets sod-db-password + sod-hcm-credentials DB admin password and HCM API credentials. Retrieved via Resource Principals at function runtime.
Step 4.1 OCI Object Storage Creates bucket with Emit Object Events ✓ landing-zone-assessment Upload landing zone. Every file write fires an OCI Event that triggers sod-enqueuer automatically.
Step 4.2 OCI Autonomous Database Creates ATP in private subnet (Private Endpoint Only) sod-assessment-adb → sod-private-subnet Central data store for all SoD assessment data. Only accessible from inside the private subnet.
Step 5.1 / 5.2 OCI Queue Service Creates DLQs first sod-ingestion-dlq + sod-connector-dlq Safety nets for failed messages. Must be created before primary queues (primary queues reference DLQ OCID).
Step 5.3 sod-ingestion-queue Primary queue (300s visibility, DLQ: sod-ingestion-dlq) sod-enqueuer → sod-processor sod-enqueuer puts messages here. sod-processor dequeues them to load audit data into ATP.
Step 5.4 sod-connector-queue Primary queue (600s visibility, DLQ: sod-connector-dlq) sod-manual-trigger → sod-connector-processor sod-manual-trigger enqueues HCM sync jobs. sod-connector-processor dequeues and executes them.
Step 6.1 / 6.2 OCI IAM Dynamic Groups match Functions + Database by resource type sod-functions-dg + sod-adb-dg Automatically includes all functions and the ATP DB in the compartment as identifiable Resource Principals.
Step 6.3 / 6.4 sod-functions-policy + sod-adb-policy IAM policies grant: read secrets · manage objects · use queues · invoke functions All OCI resources in compartment Without policies, every resource call returns 403. These policies define exactly what each resource can do.
Step 7.1 / 7.2 Local Docker CLI docker login → build → push images to OCIR sod-repo (OCIR) 7 container images packaged and stored privately. OCI Functions pulls the correct image when invoked.
Step 7.3 OCI Functions Application binds to private subnet sod-app → sod-private-subnet All 7 functions inherit private subnet network config — can reach ATP DB and OCI services internally.
Step 7.4 sod-app 7 functions created with OCIR image paths + memory + timeout sod-api-receiver, sod-enqueuer, sod-processor, sod-manual-trigger, sod-connector-processor, sod-run-status, sod-list-connectors Each function definition points to its OCIR image. OCI cold-starts a container on each invocation.
Step 7.5 sod-app Configuration Application-level environment variables All 7 functions (inherited) DB_SECRET_OCID, INGESTION_QUEUE_OCID, BUCKET_NAME etc. — no hardcoded values in function code.
Step 8.1 OCI API Management Creates Public gateway in public subnet sod-api-gateway → sod-public-subnet Public IP assigned by OCI. Internet clients send HTTPS requests to this gateway hostname.
Step 8.2 sod-api-gateway Creates deployment with prefix /sod/v1 sod-api-v1-routes Versioned route configuration. All 4 routes are defined inside this deployment.
Step 8.3 / 8.4 sod-api-v1-routes 4 routes: POST /ingest, POST /connectors/run, GET /runs/{id}/status, GET /connectors sod-api-receiver, sod-manual-trigger, sod-run-status, sod-list-connectors Each route maps an HTTP method + path to the backend function that handles it.
Step 9.1 OCI Events Service Creates rule: Object Create on landing-zone-assessment bucket sod-object-upload-trigger Fires automatically on every file upload. Two conditions ensure only uploads to this specific bucket trigger it.
Step 9.2 sod-object-upload-trigger Action: invoke function sod-enqueuer (in sod-app) The event fires sod-enqueuer with the event payload. Enqueuer puts a message on the ingestion queue. Processing begins.
Step 10.1 OCI Notifications Creates topic + email subscription sod-alert-topic → operator email Central alert channel. All 6 monitoring alarms publish to this topic when triggered.
Step 10.2 OCI Monitoring 6 alarm definitions across oci_queue, oci_functions, oci_autonomous_database namespaces sod-alert-topic Real-time alerting on DLQ depth, function errors, queue backlog, DB CPU and storage saturation.
