Networking — VCN, Gateways & Subnets

This is the foundation of the entire platform. Before any database, function, or storage resource can communicate, they all need to live inside a defined network boundary. This page provisions that boundary — the Virtual Cloud Network — and wires up every routing path the platform needs to operate securely.

ⓘ Prerequisites — Before You Start

You need an active OCI Tenancy and at least one Compartment created under it. Every resource on this page must be placed inside the same compartment. You cannot skip Step 1 — the gateways and route tables created here are referenced by subnets in Step 2.

Step 1 — Create VCN & Attach Gateways

Think of the VCN as a private data center you own inside OCI. It defines the IP address space your resources can use. The three gateways you attach to it control how traffic flows in three different directions: in from the internet, out to the internet (from private resources), and across to internal Oracle services like Object Storage and the ATP Database.

STEP 1.1
Networking Virtual Cloud Networks Create VCN

What you're doing: Creating the master network boundary for the entire SoD platform. The CIDR block 10.0.0.0/16 gives you 65,536 private IP addresses — far more than needed, but using a /16 gives ample room to add more subnets later without reconfiguring. The DNS label sodvcn allows OCI resources to resolve each other by hostname inside the network instead of using raw IPs.

How to do it: Navigate to Networking → Virtual Cloud Networks in the OCI Console and click Create VCN. Fill in the values below and click Create.

VCN Name
sod-network-vcn
IPv4 CIDR Block
10.0.0.0/16
DNS Label
sodvcn
STEP 1.2
sod-network-vcn Internet Gateways Create Internet Gateway

What you're doing: The Internet Gateway is the single entry/exit point for all internet-facing traffic in the VCN. Without it, nothing from the public internet can reach the API Gateway endpoint you'll create later. It works at the VCN level — you enable it here, then in Step 2 you tell specific subnets to use it via a route rule.

How to do it: Open sod-network-vcn, scroll to Resources on the left panel, click Internet Gateways, then Create Internet Gateway.

Gateway Name
sod-internet-gateway
Status on Creation
Enabled
STEP 1.3
sod-network-vcn NAT Gateways Create NAT Gateway

What you're doing: The NAT (Network Address Translation) Gateway solves an important security challenge — the serverless functions and database live in a private subnet with no public IPs, so they cannot directly reach external APIs like Workday or Oracle HCM. The NAT Gateway acts as an intermediary: private resources route outbound traffic through it, which makes the requests appear to come from the NAT's public IP. External services respond back, but cannot initiate connections inward. Choose Ephemeral Public IP since the exact public IP doesn't matter — only outbound requests are made.

How to do it: In the VCN details, click NAT Gateways then Create NAT Gateway.

Gateway Name
sod-nat-gateway
Public IP Type
Ephemeral Public IP
STEP 1.4
sod-network-vcn Service Gateways Create Service Gateway

What you're doing: The Service Gateway routes traffic to Oracle-managed services (Object Storage, ATP Database, OCI Queue) entirely over Oracle's private internal backbone — bypassing the public internet completely. This means when the ATP database calls DBMS_CLOUD to read a file from Object Storage, that data never touches the public internet. This reduces latency, eliminates egress costs, and keeps sensitive audit data completely off public networks. Select All Services in Oracle Services Network to cover every OCI managed service in the region.

How to do it: Click Service Gateways then Create Service Gateway.

Gateway Name
sod-service-gateway
Services
All Services in Oracle Services Network

Step 2 — Create Route Tables & Subnets

Route Tables are like the GPS routing map for each subnet — they tell the subnet: "if traffic is going to this destination, send it through this gateway". Subnets are the actual IP segments that resources sit in. You'll create one route table + subnet pair for the public zone (the API Gateway) and one for the private zone (Functions + Database).

STEP 2.1
sod-network-vcn Route Tables Create Public Route Table

What you're doing: Creating a routing rulebook for the public subnet. The single rule says: "send all internet-bound traffic (0.0.0.0/0) through the Internet Gateway". This is what allows the API Gateway to be reachable from a browser or client application anywhere in the world. The API Gateway is the only resource that will ever sit in the public subnet — everything else stays private.

How to do it: Click Route Tables then Create Route Table. Add one Route Rule before saving.

Route Table Name
sod-public-route-table
Route Rule → Destination CIDR
0.0.0.0/0
Route Rule → Target Type
Internet Gateway
Route Rule → Target
sod-internet-gateway
STEP 2.2
sod-network-vcn Route Tables Create Private Route Table

What you're doing: The private route table has two rules working together. Rule 1 handles outbound traffic to the public internet (e.g. calling an HCM API) — it routes those requests through the NAT Gateway. Rule 2 handles traffic destined for OCI services like Object Storage — it routes those through the Service Gateway, keeping them entirely on Oracle's private network. Together these two rules mean: private resources can reach everything they need, without ever being directly reachable from the internet.

How to do it: Create a second Route Table and add two Route Rules before saving.

Route Table Name
sod-private-route-table
Rule 1 — Destination CIDR / Target Type / Target
0.0.0.0/0 → NAT Gateway → sod-nat-gateway
Rule 2 — Destination Service / Target Type / Target
All Oracle Services in Oracle Services Network → Service Gateway → sod-service-gateway
STEP 2.3
sod-network-vcn Subnets Create Public Subnet

What you're doing: Carving out a 256-address IP segment (10.0.1.0/24) inside the VCN dedicated to public-facing resources. Only the API Gateway will ever be placed here. By assigning it the public route table, any resource deployed in this subnet automatically has a path to the internet. The DNS label sodpub allows OCI-internal hostname resolution.

How to do it: Click Subnets then Create Subnet. Set the subnet type to Regional and access to Public.

Subnet Name
sod-public-subnet
CIDR Block
10.0.1.0/24
Subnet Access
Public Subnet
Route Table
sod-public-route-table
DNS Label
sodpub
STEP 2.4
sod-network-vcn Subnets Create Private Subnet

What you're doing: Creating the secured internal zone where all sensitive workloads run — the serverless functions, the ATP database. By selecting Private Subnet and enabling Prohibit Public IP Addresses, OCI prevents any resource in this subnet from ever being assigned a public IP. An attacker cannot reach these resources directly from the internet — all access must go through the API Gateway in the public subnet first. The private route table gives resources here outbound internet access via NAT when needed.

How to do it: Create a second Subnet. Ensure you check Prohibit Public IP Addresses.

Subnet Name
sod-private-subnet
CIDR Block
10.0.2.0/24
Subnet Access
Private Subnet (Prohibit Public IPs)
Route Table
sod-private-route-table
DNS Label
sodpriv
✓ Checkpoint — Networking Complete

You should now have: 1 VCN (sod-network-vcn), 3 Gateways (Internet + NAT + Service), 2 Route Tables (public + private), and 2 Subnets (public 10.0.1.0/24 + private 10.0.2.0/24). All future resources will be provisioned into one of these two subnets. Continue to Step 3.
