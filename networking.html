Phase 1

Virtual Cloud Network & Subnets Setup

Configures the underlying network boundary, regional subnets, and gateway routing targets.

Networking Configuration Inputs

The table below summarizes the exact values to input during network provisioning:

Resource Configuration Inputs Reference Resource in HCL Design Rationale & Explanation
Virtual Cloud Network (VCN) Name: `sod-network-vcn`
CIDR: `10.0.0.0/16`
DNS: `sodvcn`		 oci_core_vcn.sod_vcn Establishes the core network boundary. A `/16` block allows up to 65,536 private IPs, providing ample range to isolate applications and stage staging subnets safely.
Internet Gateway Name: `sod-internet-gateway`
Status: Enabled		 oci_core_internet_gateway.sod_ig Provides a path for public ingress requests to route into the public API Gateway subnet. Unused by the database.
NAT Gateway Name: `sod-nat-gateway` oci_core_nat_gateway.sod_nat Allows private functions to safely initiate outgoing requests (e.g. fetching external HCM API data) without exposing private database IPs to the public web.
Service Gateway Name: `sod-service-gateway`
Services: All Services in Oracle Services Network		 oci_core_service_gateway.sod_sg Routes storage and database operations internally over Oracle's private backbone, completely bypassing public internet transit for maximum security.
Public Subnet Name: `sod-public-subnet`
CIDR: `10.0.1.0/24`
Route Table: `sod-public-route-table`		 oci_core_subnet.sod_public_subnet Hosts public endpoints (the OCI API Gateway). Demarcated from backend business layers to act as a secure public load-balancing landing zone.
Private Subnet Name: `sod-private-subnet`
CIDR: `10.0.2.0/24`
Route Table: `sod-private-route-table`		 oci_core_subnet.sod_private_subnet Houses private execution layers (Functions, ATP Database). Direct public internet routing is prohibited to block network scanning attempts.

Step 1 — Configure VCN & Gateways

Initialize the core network domain boundary and add gateways to allow internal/external routing pathways.

STEP 1.1
Networking Virtual Cloud Networks Create VCN

Enter VCN attributes and enable DNS label resolution configuration in your compartment.

VCN Name
sod-network-vcn
IPv4 CIDR Block
10.0.0.0/16
DNS Label
sodvcn
STEP 1.2
VCN Details Resources Internet Gateways Create Gateway

Create the internet gateway to allow public ingress and egress routing bounds.

Gateway Name
sod-internet-gateway
STEP 1.3
VCN Details Resources NAT Gateways Create Gateway

Provision a NAT Gateway with an ephemeral public IP for safe private-subnet outbound calls.

Gateway Name
sod-nat-gateway
Public IP Type
Ephemeral Public IP
STEP 1.4
VCN Details Resources Service Gateways Create Gateway

Link all regional private Oracle services (Object Storage, ATP) to the Service Gateway.

Gateway Name
sod-service-gateway
Target Service List
All Services in Region
Why We Do It

Creating separate subnets isolates the public network space (where the API Gateway resides) from the private network space (where OCI Functions and the Autonomous Database execute), completely blocking direct scanner attacks from the internet.

Step 2 — Configure Route Tables & Subnets

Configure access rules to govern subnet routing scopes and allocate regional addresses.

STEP 2.1
VCN Details Route Tables Create Route Table (Public)

Define route rules for internet ingress endpoints routing via the Internet Gateway.

Route Table Name
sod-public-route-table
Rule Destination
0.0.0.0/0
Target Resource
sod-internet-gateway
STEP 2.2
VCN Details Route Tables Create Route Table (Private)

Set up private routing allowing outgoing traffic via NAT Gateway and internal storage connections via Service Gateway.

Route Table Name
sod-private-route-table
Rule 1 (Outbound NAT)
Destination: 0.0.0.0/0 ➔ Target: sod-nat-gateway
Rule 2 (Internal Service Gateway)
Destination: All Services ➔ Target: sod-service-gateway
STEP 2.3
VCN Details Subnets Create Subnet

Provision regional subnets in the VCN, ensuring the database subnet is strictly private.

Public Subnet Properties
Name: sod-public-subnet | CIDR: 10.0.1.0/24 | Access: Public | Route Table: sod-public-route-table | DNS: sodpub
Private Subnet Properties
Name: sod-private-subnet | CIDR: 10.0.2.0/24 | Access: Private (Prohibit Public IPs) | Route Table: sod-private-route-table | DNS: sodpriv
Why We Do It

Using Route Tables allows you to configure rules so private functions can access external APIs (like Workday APIs) through the NAT Gateway, and access internal storage systems privately through the Service Gateway, keeping database-to-storage traffic off the public internet.
