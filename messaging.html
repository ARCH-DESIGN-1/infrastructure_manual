Messaging — OCI Queue Pairs & Dead Letter Queues

Provisions decoupled asynchronous messaging queues to buffer file ingestion events and external connector triggers, with automatic retry and dead-letter routing on failure.

ⓘ Important — Creation Order

You must create the Dead Letter Queues (DLQs) first before creating the primary queues. The primary queues reference the DLQ OCIDs during setup. Creating them in reverse order will fail with a validation error.

Step 5 — Create DLQ Pairs then Primary Queues

OCI Queue is a managed messaging service that decouples the file upload event from the heavy data processing work. Instead of the upload triggering direct database writes (which could fail and lose data), it puts a lightweight message into a queue. The processor function then dequeues that message and does the heavy lifting. If processing fails, the message is retried automatically up to a configurable number of times. If all retries fail, the message is moved to the Dead Letter Queue (DLQ) for investigation — no data is lost.

Queue Name Type Visibility Timeout Max Delivery Attempts DLQ Target
sod-ingestion-dlq Dead Letter Queue 30 seconds N/A — (is a DLQ)
sod-connector-dlq Dead Letter Queue 30 seconds N/A — (is a DLQ)
sod-ingestion-queue Primary Queue 300 seconds 3 sod-ingestion-dlq
sod-connector-queue Primary Queue 600 seconds 3 sod-connector-dlq
STEP 5.1
Developer Services Queues Create Ingestion DLQ (First)

What you're doing: A Dead Letter Queue (DLQ) is a safety net — it's just a regular queue used to capture messages that failed all processing retries. You create it first because the primary queue (Step 5.3) will reference this DLQ by OCID during setup. The DLQ itself has no DLQ assignment. When a file ingestion message fails 3 times in a row, it lands here so operators can investigate it without the message being lost forever.

How to do it: Navigate to Developer Services → Queues and click Create Queue.

STEP 5.2
Developer Services Queues Create Connector DLQ (Second)

What you're doing: Creating the DLQ for the connector pipeline. HCM API sync failures (network timeouts, auth expiry, rate limits) will be moved here after all retries are exhausted. Separating the connector DLQ from the ingestion DLQ means you can monitor and alert on each pipeline independently — a spike in the connector DLQ points to an HCM API issue; a spike in the ingestion DLQ points to a data format problem.

How to do it: Repeat the same steps as 5.1 with the name below.

sod-connector-dlq

30
Message Retention (seconds)
86400
Dead Letter Queue
None — leave blank
STEP 5.3
Developer Services Queues Create Primary Ingestion Queue

What you're doing: Creating the main ingestion pipeline queue. The Visibility Timeout of 300 seconds is the critical setting here — when the processor function dequeues a message, OCI hides that message from other consumers for 300 seconds (5 minutes). This gives the function time to download the file, parse it, and insert all rows into the database. If the function finishes before 300s, it deletes the message. If it crashes, the timeout expires and the message becomes visible again for retry. The 300s value is chosen because even a 50,000-row CSV file should load in under 4 minutes.

How to do it: Create a new queue and set the Dead Letter Queue to sod-ingestion-dlq.

Queue Name
sod-ingestion-queue
Visibility Timeout (seconds)
300
Message Retention (seconds)
86400
Dead Letter Queue
sod-ingestion-dlq
Max Delivery Attempts
3
STEP 5.4
Developer Services Queues Create Primary Connector Queue

What you're doing: Creating the connector sync queue with a longer 600-second visibility timeout. HCM connector syncs are much heavier operations — they authenticate against an external API, paginate through potentially thousands of employee records, download them, transform the data, and write files to Object Storage. 10 minutes gives even the largest HCM tenants enough processing window. Setting Max Delivery Attempts to 3 means a temporary HCM API outage will get up to 2 automatic retries before the job is moved to the DLQ for operator review.

How to do it: Create another queue and set the Dead Letter Queue to sod-connector-dlq.

✓ Checkpoint — Queues Complete

You should now have 4 queues total: 2 DLQs (sod-ingestion-dlq, sod-connector-dlq) and 2 primary queues (sod-ingestion-queue, sod-connector-queue). Save the queue OCIDs — needed for IAM policies in Step 6.
