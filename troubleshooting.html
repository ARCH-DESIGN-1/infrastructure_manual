🔧 Troubleshooting & Debugging

A real-world debugging runbook that documents every diagnostic command used to identify and resolve issues during the initial deployment of the SoD platform. Use this as a reference whenever an API endpoint is timing out, returning errors, or behaving unexpectedly.

Issue 1 — API endpoint keeps loading (gateway timeout)

Symptom: Calling GET /sod/v1/connectors never returns — the browser or curl hangs indefinitely until the API Gateway returns a 504 or 502.
Root cause found: 3 misconfigurations in the sod-app environment config + missing database ingress rule on the private subnet security list.

DEBUG 1

Check current sod-app environment configuration. The very first step is to verify what environment variables the Functions Application actually has set — misconfigured values here are the most common cause of timeouts.

# Get all config key-value pairs currently set on sod-app oci fn application get \ --application-id ocid1.fnapp.oc1.ap-mumbai-1.amaaaaaarnlqjtiardxrncpwevjbnpjrjthsj36buvaaladyeipystjp7toa \ --query 'data.config'
# What was returned (truncated): { "ATP_CONNECTION_STRING": "(description= (retry_count=20)...)", "BUCKET_NAME": "sod-landing-zone", "CONNECTOR_QUEUE_OCID": "ocid1.queue...ullfbf5yzptq-dlq", <-- ❌ This is a Dead Letter Queue! "DB_SECRET_OCID": "ocid1.autonomousdatabase...", <-- ❌ This is the ADB OCID, not a Vault Secret! "INGESTION_QUEUE_OCID": "ocid1.queue...cjviakcgmt2mftq-dlq", <-- ❌ This is a Dead Letter Queue! "DB_USER": "sod_admin", "REGION": "ap-mumbai-1" }
❌ 3 misconfigured values found
VariableBad Value (was set to)Problem
DB_SECRET_OCID ocid1.autonomousdatabase... Was set to the ADB database resource OCID instead of the Vault Secret OCID. Causes SecretsClient.get_secret_bundle() to throw a 404/AuthZ error before the DB connection is even attempted.
INGESTION_QUEUE_OCID ...mftq-dlq Pointing to the Dead Letter Queue instead of the main ingestion queue. Functions trying to put_messages() here will be rejected.
CONNECTOR_QUEUE_OCID ...ptq-dlq Same DLQ issue for the connector queue. The -dlq suffix is the dead letter variant — always use the primary queue OCID.
DEBUG 2

List all Vault Secrets to get the correct DB_SECRET_OCID. Use this command to find the OCID of the sod-db-password secret that stores the database password.

# List all secrets in the compartment to identify the correct one oci vault secret list \ --compartment-id ocid1.compartment.oc1..aaaaaaaauj3rfwoniy2hwww43z6k5gdg377ww6wsgyfpo2x4ztqle5hdrtta
# Relevant secret found: { "id": "ocid1.vaultsecret.oc1.ap-mumbai-1.amaaaaaarnlqjtiaayskhu6ykny3jky42inswiqkuwp57uwkrkfvy7uxmfgq", "secret-name": "sod-db-password", <-- ✅ This is the correct OCID for DB_SECRET_OCID "lifecycle-state": "ACTIVE" }
✅ Correct Vault Secret OCID identified
DEBUG 3

Apply all config fixes in one command. Update the application config using the OCI CLI, setting the correct Vault Secret OCID, correct queue OCIDs (without -dlq), and all other required values.

# Fix all 3 misconfigurations in one update command oci fn application update \ --application-id ocid1.fnapp.oc1.ap-mumbai-1.amaaaaaarnlqjtiardxrncpwevjbnpjrjthsj36buvaaladyeipystjp7toa \ --config '{ "DB_SECRET_OCID": "ocid1.vaultsecret.oc1.ap-mumbai-1.amaaaaaarnlqjtiaayskhu6ykny3jky42inswiqkuwp57uwkrkfvy7uxmfgq", "ATP_CONNECTION_STRING": "(description= (retry_count=20)(retry_delay=3)(address=(protocol=tcps)(port=1522)(host=fgfjmbzr.adb.ap-mumbai-1.oraclecloud.com))(connect_data=(service_name=gd8ea41c6489263_sodadb_high.adb.oraclecloud.com))(security=(ssl_server_cert_dn=\"CN=adb.ap-mumbai-1.oraclecloud.com, O=Oracle Corporation, L=Redwood City, ST=California, C=US\")))", "BUCKET_NAME": "sod-landing-zone", "DB_USER": "sod_admin", "REGION": "ap-mumbai-1", "INGESTION_QUEUE_OCID": "ocid1.queue.oc1.ap-mumbai-1.amaaaaaarnlqjtiang4yzws2foqbzk24tk5rwsnd5dfs4cjviakcgmt2mftq", "CONNECTOR_QUEUE_OCID": "ocid1.queue.oc1.ap-mumbai-1.amaaaaaarnlqjtiasp5xbjhcroyhppoeev4hhtkx2gmbvmx6ullfbf5yzptq" }'
# CLI will prompt for confirmation — type y and press Enter: WARNING: Updates to config ... will replace any existing values. Are you sure? [y/N]: y # On success, it returns the updated application JSON with "time-updated" refreshed.
✅ Config updated — changes take effect on next invocation

Issue 2 — Function hangs even after config fix (DB connection blocked)

Symptom: After fixing the config, requests still time out. Function logs (if enabled) show the code reaching oracledb.connect() but never returning.
Root cause found: The private subnet Security List was missing an ingress rule allowing TCP traffic on port 1522 (the ATP TLS port).

DEBUG 4

Identify the Autonomous Database private endpoint and subnet. Confirm the ADB is on a private endpoint and which subnet it belongs to, so you know which security list to update.

# List all Autonomous Databases in the compartment oci db autonomous-database list \ --compartment-id ocid1.compartment.oc1..aaaaaaaauj3rfwoniy2hwww43z6k5gdg377ww6wsgyfpo2x4ztqle5hdrtta
# Key fields from the output: "db-name": "sodadb" "private-endpoint": "fgfjmbzr.adb.ap-mumbai-1.oraclecloud.com" "private-endpoint-ip": "10.0.2.116" <-- ADB IP is in the private subnet "subnet-id": "ocid1.subnet.oc1.ap-mumbai-1.aaaaaaaa6tduo2hx45czbymqhyxpngv2l2uotjfyvr6herp35oj26arbxkwa" "nsg-ids": null <-- No NSGs attached to ADB itself
DEBUG 5

Get the private subnet's Security List OCID. The subnet references which security list is applied to it — we need this OCID to inspect and update the rules.

# Fetch the subnet details to find its security-list-ids oci network subnet get \ --subnet-id ocid1.subnet.oc1.ap-mumbai-1.aaaaaaaa6tduo2hx45czbymqhyxpngv2l2uotjfyvr6herp35oj26arbxkwa
# Key field from the output: "display-name": "sod-private-subnet" "cidr-block": "10.0.2.0/24" "security-list-ids": [ "ocid1.securitylist.oc1.ap-mumbai-1.aaaaaaaaunw2cqap3kjg5wka4dpipbayc6un4zrljt6yivtaynp2oqc6h6tq" ]
DEBUG 6

Inspect the Security List to verify all ingress rules. Fetch the full security list to see if port 1522 is allowed. If missing, this is the reason the database connection hangs.

# Get the full ingress and egress rule list for the private subnet security list oci network security-list get \ --security-list-id ocid1.securitylist.oc1.ap-mumbai-1.aaaaaaaaunw2cqap3kjg5wka4dpipbayc6un4zrljt6yivtaynp2oqc6h6tq
# After the fix was applied, the ingress-security-rules array contained: { "description": "Allow OCI Functions in private subnet to connect to Autonomous Database", "source": "10.0.2.0/24", <-- From private subnet CIDR "protocol": "6", <-- TCP "tcp-options": { "destination-port-range": { "min": 1522, "max": 1522 } <-- ✅ ADB TLS port } }
✅ Port 1522 ingress rule confirmed in Security List

🔧 How to add the missing rule via OCI Console

  1. Navigate to Networking → Virtual Cloud Networks → sod-network-vcn → Security Lists
  2. Open the security list associated with sod-private-subnet
  3. Click Add Ingress Rules and fill in:
    • Source CIDR: 10.0.2.0/24
    • IP Protocol: TCP
    • Destination Port Range: 1522
    • Description: Allow OCI Functions in private subnet to connect to Autonomous Database

DEBUG 9

Get detailed configuration of a single function (e.g. timeout, memory). Use this to confirm the function's memory allocation and timeout are not too low, which would cause premature kills.

# First get the function OCID by listing functions (DEBUG 8), then: oci fn function get --function-id <FUNCTION_OCID>
# Key fields to verify: "memory-in-mbs": 256 <-- Minimum 256 MB for DB-connected functions "timeout-in-seconds": 300 <-- Max 300s; must be less than API GW timeout (60s default) "lifecycle-state": "ACTIVE" "invoke-endpoint": "https://..."

⚠️ If timeout-in-seconds is too high

OCI API Gateway has a hard 60-second timeout. If your function timeout is set higher, the gateway will return 504 before your function finishes. Update it with:

oci fn function update --function-id <OCID> --timeout-in-seconds 60

Issue 6 — Verify IAM Dynamic Group Policies

Symptom: Functions fail with NotAuthenticatedException or Authorization failed in logs when accessing Vault, Queue, or Object Storage — even though config is correct.
Action: Verify the Dynamic Group and IAM Policies that grant the functions Resource Principal access.

DEBUG 12

List all IAM Dynamic Groups in the tenancy. Confirms sod-functions-dg exists and its matching rule covers your Functions Application.

# List all dynamic groups (tenancy-level resource) oci iam dynamic-group list --all
# Look for sod-functions-dg and verify its matching-rule: "name": "sod-functions-dg" "matching-rule": "ALL {resource.type = 'fnfunc', resource.compartment.id = 'ocid1.compartment...'}" "lifecycle-state": "ACTIVE" # If missing, create it: oci iam dynamic-group create \ --name "sod-functions-dg" \ --description "Dynamic group for SoD platform OCI Functions" \ --matching-rule "ALL {resource.type = 'fnfunc', resource.compartment.id = 'ocid1.compartment.oc1..aaaaaaaauj3rfwoniy2hwww43z6k5gdg377ww6wsgyfpo2x4ztqle5hdrtta'}"
DEBUG 13

List all IAM Policies in the compartment. Confirms the policies granting sod-functions-dg access to Vault, Queue, Object Storage and ADB exist.

# List all policies in the compartment oci iam policy list \ --compartment-id ocid1.compartment.oc1..aaaaaaaauj3rfwoniy2hwww43z6k5gdg377ww6wsgyfpo2x4ztqle5hdrtta \ --all
# Each policy's "statements" array must include rules like: "Allow dynamic-group sod-functions-dg to read secret-bundles in compartment sod-network" "Allow dynamic-group sod-functions-dg to use queues in compartment sod-network" "Allow dynamic-group sod-functions-dg to manage objects in compartment sod-network" "Allow dynamic-group sod-functions-dg to use autonomous-databases in compartment sod-network" # Missing any of these = 403 Authorization errors in function logs

🔧 How to create a missing policy

Run this once to add all required permissions in one policy:

oci iam policy create \ --compartment-id ocid1.compartment.oc1..aaaaaaaauj3rfwoniy2hwww43z6k5gdg377ww6wsgyfpo2x4ztqle5hdrtta \ --name "sod-functions-policy" \ --description "Grants SoD functions access to required OCI services" \ --statements '[ "Allow dynamic-group sod-functions-dg to read secret-bundles in compartment sod-network", "Allow dynamic-group sod-functions-dg to use queues in compartment sod-network", "Allow dynamic-group sod-functions-dg to manage objects in compartment sod-network", "Allow dynamic-group sod-functions-dg to use autonomous-databases in compartment sod-network", "Allow dynamic-group sod-functions-dg to read buckets in compartment sod-network" ]'

Issue 9 — Test the API endpoint directly with curl

Symptom: Not sure if the issue is the browser, the gateway, or the function itself.
Action: Use curl with verbose output to see exact HTTP status codes, response times, and headers.

DEBUG 17

Call each API endpoint with curl and measure response time. The -w timing flag reveals where the delay occurs — DNS, TCP connect, TLS, or server response.

# Test GET /sod/v1/connectors (list connectors) curl -v -w "\n\nTime DNS: %{time_namelookup}s\nTime Connect: %{time_connect}s\nTime TLS: %{time_appconnect}s\nTime TTFB: %{time_starttransfer}s\nTotal: %{time_total}s\n" \ "https://fus5pxhueodg6a7moyj4mryvmy.apigateway.ap-mumbai-1.oci.customer-oci.com/sod/v1/connectors" # Test POST /sod/v1/trigger (manual trigger) curl -v -X POST \ -H "Content-Type: application/json" \ -d '{"connectorId": "oracle-hcm-fusion", "runMode": "FULL", "triggeredBy": "debug-test"}' \ "https://fus5pxhueodg6a7moyj4mryvmy.apigateway.ap-mumbai-1.oci.customer-oci.com/sod/v1/trigger" # Test GET /sod/v1/runs/{runId}/status curl -v "https://fus5pxhueodg6a7moyj4mryvmy.apigateway.ap-mumbai-1.oci.customer-oci.com/sod/v1/runs/<RUN_ID>/status"
# Interpreting the timing output: Time DNS: < 0.1s ✅ Normal Time Connect: < 0.3s ✅ Normal (gateway reachable) Time TTFB: > 30s ❌ Function is hanging (DB/networking issue) Total: ~60s ❌ API Gateway hit its timeout limit (504) # Expected healthy response: HTTP 200 → Time TTFB < 5s on warm invocation, < 15s on cold start

Issue 12 — Verify Vault secret value is correct

Symptom: Functions retrieve the secret successfully but still fail DB login — the password stored in Vault may be wrong or stale.
Action: Read and decode the current secret value to verify it matches the ADB admin password.

DEBUG 20

Read the current Vault secret content and decode it. Retrieves the base64-encoded secret bundle and decodes it so you can verify the password is exactly what you expect.

# Retrieve the secret bundle and decode it in one pipeline oci secrets secret-bundle get \ --secret-id ocid1.vaultsecret.oc1.ap-mumbai-1.amaaaaaarnlqjtiaayskhu6ykny3jky42inswiqkuwp57uwkrkfvy7uxmfgq \ --query 'data."secret-bundle-content".content' \ --raw-output | base64 --decode
# Output: the plaintext password stored in Vault MyDatabasePassword123! # Verify this exactly matches the ADMIN/sod_admin password set during ADB creation # If wrong, update the secret with: echo -n "NewPassword!" | base64 # Then update the secret version in the OCI Console: Vault → Secrets → sod-db-password → Create Secret Version

💡 General Debugging Approach — in order
  1. Check env vars first (DEBUG 1) — wrong OCIDs are the #1 silent failure cause.
  2. Enable OCI Function Logs (DEBUG 19) — Python tracebacks reveal exactly which line fails.
  3. Use curl with timing (DEBUG 17) — if TTFB > 30s, it's networking or DB, not code.
  4. Verify Security List port 1522 (DEBUG 6) — required even on same subnet as ADB.
  5. Verify IAM policies (DEBUG 13) — missing rules cause silent 403s in function logs.
  6. Queue OCIDs must never end in -dlq — DLQs are receive-only overflow channels.
  7. ADB must be AVAILABLE (DEBUG 16) — serverless ADB may auto-stop after inactivity.
  8. After any config change, next cold-start picks it up — no redeploy needed.
