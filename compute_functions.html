OCIR & Functions — Container Registry, Application & Serverless Deployment

OCI Functions is the serverless compute engine of the SoD platform. Each function is a small, self-contained Python container image that runs on-demand — you only pay when it executes. This page covers packaging those containers, pushing them to Oracle's private registry (OCIR), and wiring them up inside OCI as deployable serverless functions.

ⓘ Prerequisites — Before You Start

You need three things ready before starting this step:

  1. Docker Desktop installed and running on your local machine
  2. SoD platform source code cloned from the repository — each function has its own directory with a Dockerfile
  3. OCI Tenancy Namespace — find it at: Profile icon (top right) → Tenancy → Object Storage Namespace field

How OCI Functions Works — Overview

The deployment flow has four stages that must be done in order:

Deployment Order

1. OCIR Repository (storage for images) → 2. Build & Push Docker images → 3. Functions Application (logical grouping container) → 4. Function definitions (individual function configs pointing to images)

Function Deployment Parameters

Click any row to view the full source code for that function. Use these memory and timeout values when creating each function in Step 7.4:

Function Name Memory (MB) Timeout (s) Trigger Role in the Platform
sod-api-receiver 25660 API Gateway POST /ingest Entry point for client file uploads. Validates the request, uploads the file to Object Storage, creates a run record in the DB, and returns a Run ID to the client immediately.
sod-enqueuer 12860 OCI Events (bucket upload) Fires automatically when a file lands in the bucket. Reads the event payload (file name, bucket) and pushes a message onto the ingestion queue to kick off processing.
sod-processor 256300 OCI Queue (message poll) The core data loader. Dequeues messages, downloads the file from Object Storage, parses CSV/JSON rows, and inserts them into the ATP staging tables. 300s timeout handles large multi-megabyte audit files.
sod-manual-trigger 25630 API Gateway POST /connectors/run Lets operators manually trigger a specific HCM connector sync on-demand. Validates the request and puts a command message onto the connector queue.
sod-connector-processor 256600 OCI Queue (connector queue) The HCM integration worker. Dequeues connector commands, authenticates against the target HCM API (Workday, Oracle HCM Cloud), downloads the employee/role data, and deposits the file into the landing bucket. 600s timeout for slow large downloads.
sod-run-status 25630 API Gateway GET /runs/{runId}/status Lightweight status polling function. Queries the database run record for the given runId and returns the current state (QUEUED, PROCESSING, COMPLETE, FAILED) with counts.
sod-list-connectors 25630 API Gateway GET /connectors Returns the list of active HCM connector configurations registered in the database, so client applications know which systems are available for sync.

Step 7 — Deploy Container Registry & Functions

STEP 7.1
Developer Services Container Registry (OCIR) Create Repository

What you're doing: OCIR (Oracle Cloud Infrastructure Registry) is Oracle's private Docker image registry — think of it as a private Docker Hub that lives inside your OCI tenancy. You need a repository (a named folder) before you can push images to it. Setting it to Private means only authenticated OCI users and functions with the correct IAM policy can pull images. This is where all 7 function container images will be stored and versioned.

How to do it: Navigate to Developer Services → Container Registry, click Create Repository, and fill in the values below.

Repository Name
sod-repo
Access
Private
STEP 7.2
Local Terminal Docker CLI Login → Build → Tag → Push (×7)

What you're doing: Authenticating Docker to the OCIR registry so it can upload images to your private repository. The username format is non-obvious — OCI requires you to prefix your regular OCI username with the tenancy namespace separated by a forward slash. Your password is an OCI Auth Token (not your OCI console password) — generate one at Profile → Auth Tokens → Generate Token. Then for each of the 7 function source directories, you run docker build to create the image and docker push to upload it to OCIR.

How to do it: Run these commands from your local terminal. Replace placeholders with your actual values.

Step A — Login to OCIR (use Auth Token as password)
docker login <region-key>.ocir.io
OCIR Username Format
<tenancy-namespace>/<your-oci-username>
Step B — Build image (run from inside each function's directory)
docker build -t <region-key>.ocir.io/<tenancy-namespace>/sod-repo/<fn-name>:1.0.0 .
Step C — Push image to OCIR
docker push <region-key>.ocir.io/<tenancy-namespace>/sod-repo/<fn-name>:1.0.0
Mumbai region key (if deploying to ap-mumbai-1)
bom
Repeat build + push for all 7 function names
sod-api-receiver sod-enqueuer sod-processor sod-manual-trigger sod-connector-processor sod-run-status sod-list-connectors
STEP 7.3
Developer Services Functions Create Application

What you're doing: An OCI Functions Application is a logical grouping container — it doesn't run code itself, but it defines the shared network configuration (VCN + subnet) that all functions inside it inherit. All 7 functions will belong to this single application, meaning they all run inside the private subnet and can communicate with the ATP database, Object Storage, and queues without any additional network setup per function. Think of the Application as a namespace for your functions.

How to do it: Navigate to Developer Services → Functions → Applications and click Create Application.

Application Name
sod-app
VCN
sod-network-vcn
Subnets
sod-private-subnet
STEP 7.4
sod-app Details Functions Create Function (repeat ×7)

What you're doing: Creating the individual function definitions inside the sod-app application. Each function definition tells OCI: "when this function is invoked, pull this specific Docker image from OCIR, allocate this much memory, and time out after this many seconds". The image path you specify is the full OCIR path of the image you pushed in Step 7.2. You need to repeat this process 7 times — once per function.

How to do it: Open sod-app, click Create Function. For each function: enter the name, paste the full image path, set memory and timeout, then click Create. Repeat for all 7.

Full image path format

STEP 7.5
sod-app Details Configuration Set Application Environment Variables

What you're doing: OCI Functions reads configuration at runtime from Application-level environment variables. Instead of hardcoding connection strings and resource OCIDs inside the function code, you store them here so they can be updated without redeploying images. These variables tell each function: where the database is, which queue to use, what compartment it runs in, and what region it's in.

How to do it: In sod-app, click Configuration in the left menu. Add the following key-value pairs using the Add Key-Value Pair button. Replace placeholder values with your actual OCIDs.

DB_SECRET_OCID — OCID of the sod-db-password vault secret
ocid1.vaultsecret.oc1..<your-db-secret-ocid>
ATP_CONNECTION_STRING — ATP private endpoint hostname
sodassessdb_high
INGESTION_QUEUE_OCID — OCID of sod-ingestion-queue
ocid1.queue.oc1..<your-ingestion-queue-ocid>
CONNECTOR_QUEUE_OCID — OCID of sod-connector-queue
ocid1.queue.oc1..<your-connector-queue-ocid>
BUCKET_NAME — Landing zone bucket name
landing-zone-assessment
REGION — OCI region identifier
ap-mumbai-1
✓ Checkpoint — Functions Deployment Complete

At this point you should have: 1 OCIR Repository (sod-repo) containing 7 pushed Docker images, 1 Functions Application (sod-app) in the private subnet, 7 Function definitions with correct memory/timeout, and application environment variables configured. Proceed to Step 8 to wire the API Gateway in front of the appropriate functions.
