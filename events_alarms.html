Events & Alarms — Upload Triggers & Monitoring

Wires the bucket upload event to automatically invoke the enqueuer function, then creates a notification topic and metric alarms to alert operators on any pipeline failures.

ⓘ Prerequisites

The sod-enqueuer function from Step 7 must be deployed, and the landing-zone-assessment bucket from Step 4 must have Emit Object Events enabled. Without these, the event rule will not trigger.

Step 9 — Create Events Rule (Upload → Function Trigger)

OCI Events is a rules-based service that listens to activity across all OCI resources and fires actions in response. Without this rule, you'd need to poll the bucket for new files, which is wasteful and slow. With this rule, the moment a client uploads a file, OCI Events immediately calls the sod-enqueuer function with the event payload (bucket name, file name, time). The enqueuer then puts that info onto the ingestion queue. This is the "spark" that ignites the entire processing pipeline — fully automated, zero manual intervention.

STEP 9.1
Observability & Management Events Service Create Rule

What you're doing: Creating a rule with two conditions that must both be true: (1) the event type must be Object - Create (fired when any file is written to Object Storage), and (2) the bucketName attribute must equal landing-zone-assessment (so only uploads to your bucket trigger this, not every bucket in the tenancy). Adding the bucket filter is critical — without it, any upload to any bucket in your compartment would trigger the function.

How to do it: Navigate to Observability & Management → Events Service → Rules and click Create Rule. In the Rule Conditions section, add two conditions.

Rule Display Name
sod-object-upload-trigger
Rule Status
Enabled
Condition 1 — Event Type
Object Storage → Object - Create
Condition 2 — Attribute: bucketName
landing-zone-assessment
STEP 9.2
sod-object-upload-trigger Actions Add Action — Invoke Function

What you're doing: Adding the action that fires when the rule conditions are met. Setting the action type to Functions tells OCI Events to invoke the selected function synchronously, passing the event JSON payload as the function input. The sod-enqueuer function receives this payload, extracts the object name and bucket, and enqueues a processing message. This is the only step that wires the passive bucket event into an active processing chain.

How to do it: In the Actions section of the rule wizard, click Add Action. Select Functions as the action type.

Why We Do It

OCI Events eliminates polling infrastructure. The moment a client uploads an audit file to the bucket, the event instantly triggers sod-enqueuer, which places a message onto the ingestion queue. The processor function then dequeues and loads the data into ATP automatically.

Step 10 — Create Notification Topic & Monitoring Alarms

Create a centralized notification topic (with email subscription) then configure 6 telemetry alarms to alert operators on DLQ depth, function errors, queue backlog, and database resource exhaustion.

Alarm Name Namespace Severity Triggers When
SOD-DLQ-Depth-CRITICAL oci_queue CRITICAL Any message lands in the ingestion DLQ (indicates repeated parse failures)
SOD-Connector-DLQ-Depth oci_queue CRITICAL Any message lands in the connector DLQ (HCM API failures)
SOD-Processor-Errors oci_functions WARNING sod-processor function error count > 2 in 1 minute
SOD-Queue-Backlog oci_queue WARNING Ingestion queue depth average > 50 over 5 minutes
SOD-ADB-CPU-High oci_autonomous_database WARNING Database CPU utilization mean > 80% over 1 minute
SOD-ADB-Storage-Critical oci_autonomous_database CRITICAL Database storage utilization max > 85%
STEP 10.1
Developer Services Notifications Create Topic & Email Subscription

What you're doing: A Notification Topic is a fan-out channel for alerts. When an alarm fires, it publishes a message to this topic, and the topic delivers it to all subscribers (email, Slack webhook, PagerDuty, etc.). By creating one central topic, you can add multiple subscribers without reconfiguring alarms. After creating the topic, OCI sends a confirmation email to the address you subscribe — you must click the confirmation link in that email or alert emails will not be delivered.

How to do it: Navigate to Developer Services → Notifications → Topics, click Create Topic, then click Create Subscription inside the topic.

STEP 10.2
Observability & Management Monitoring Create Alarms (×6)

What you're doing: OCI Monitoring alarms continuously evaluate metric queries and fire when a condition is breached. Each alarm you create watches a specific metric from a specific OCI resource namespace. When the condition is true (e.g. DLQ depth > 0), OCI publishes an alert to sod-alert-topic, which emails all subscribers. You need 6 alarms: 2 for DLQ failures, 1 for function errors, 1 for queue backlog buildup, and 2 for database resource pressure. Together they give complete visibility into the health of every component in the pipeline.

How to do it: Navigate to Observability & Management → Monitoring → Alarm Definitions and click Create Alarm six times, once per row in the reference table above. For each alarm, set the Notification Destination to sod-alert-topic.

Example — SOD-DLQ-Depth-CRITICAL (Alarm 1)
Namespace: oci_queue | Metric: DeadLetterQueueSize | Interval: 1m | Statistic: Max | Threshold: > 0 | Severity: CRITICAL | Destination: sod-alert-topic
Repeat for all 6 alarms in the reference table above
